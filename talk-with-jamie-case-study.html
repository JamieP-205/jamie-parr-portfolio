Project case study

Talk With Jamie: a privacy-conscious personal chat app.

This is my most technical project so far. It combines a static frontend, Netlify Functions, account sessions, guest chats, persistent conversations, admin tools, private context retrieval and automated tests.

Overview

Talk With Jamie started as a personal chat interface, but the interesting part became the system around it. I wanted to explore how a small web app could support guest users, registered users, persistent conversations and admin tools while still being clear about what the app is and how data is handled.

What I built

  • Guest and registered chat sessions.
  • Signed HttpOnly session cookies and password hashing.
  • Persistent users, conversations, contacts and rate limits using Netlify Blobs.
  • A Netlify Functions backend with route handling, input cleaning and provider logic.
  • Private admin tools for conversation review, blocking, deletion and drafting.
  • A v2 deployment path with private context retrieval and public-safe context separation.
  • Automated checks covering syntax, site structure, authentication helpers, context boundaries and deployment safeguards.

Architecture

The project uses a static frontend with JavaScript handling the chat interface and account flow. Requests go through Netlify Functions, which act as the backend API. Storage is handled through Netlify Blobs, allowing the project to persist conversations and user records without running a traditional server.

Frontend Static HTML, CSS and JavaScript
API Netlify Functions routes
Storage Netlify Blobs
Testing Node syntax, site, unit and integration tests

Privacy and safety decisions

The project is clearly labelled as a chat experiment, and it warns users not to submit sensitive information. Raw personal exports and private source files are not committed or deployed. The v2 version uses a compact, redacted context pack rather than exposing original source material.

The important design choice is separation: public users only receive public-safe context, while deeper private context is limited to admin or trusted workflows. That made the project more complicated, but it also made it more realistic and safer.

Testing and deployment

The root app and v2 app both have automated checks. These cover JavaScript syntax, site structure, authentication helpers, signed sessions, username validation, message cleaning, context validation, retrieval boundaries and production migration safeguards.

What I learned

This project helped me understand the difference between making a page that works and designing an actual application with users, sessions, storage, privacy boundaries and deployment risk. It also gave me practice explaining technical decisions clearly, which is useful for support, admin and junior developer roles.
